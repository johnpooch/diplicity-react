# Gap 07: Per-Phase Diary and Memory Persistence

## Gap statement

Memory is AI_Diplomacy's centrepiece. Each agent keeps two diaries ‚Äî `full_private_diary`
(permanent) and `private_diary` (the context copy) ‚Äî of phase-prefixed entries
(`add_diary_entry`, `ai_diplomacy/agent.py`). Three entry kinds are generated by dedicated LLM
calls: a negotiation summary after messaging (`generate_negotiation_diary_entry`, driven by
`negotiation_diary_prompt.txt`), an order-reasoning note (`generate_order_diary_entry` ‚Äî note:
**disabled** in the main loop, `lm_game.py` guards it with `if False`), and a phase-result
reflection that compares negotiations against resolved orders to detect betrayals
(`generate_phase_result_diary_entry`, `phase_result_diary_prompt.txt`: "BETRAYALS: Who broke
their promises?"). The context diary is fed back into every prompt via
`format_private_diary_for_prompt`, and order generation gets only the latest phase's entries
(`get_latest_phase_diary_entries`, wired in `lm_game.py`). To stop unbounded growth, a
consolidation call summarizes all but the most recent year into a single
`[CONSOLIDATED HISTORY]` entry each Spring (`run_diary_consolidation`,
`ai_diplomacy/diary_logic.py`, `diary_consolidation_prompt.txt`). Memory lives in process and
is serialized to the run's JSON snapshot ‚Äî fine for a single-process harness.

We have **no memory at all**. Nothing in `service/bot/` persists anything between task runs
except the orders and chat messages the bot happens to submit through the API. Each `plan`,
`finalize`, and `reply` task re-fetches the world (`fetch_context`,
`service/bot/context/fetch.py`) and reasons from scratch; the model's stated reasoning is
thrown away (`parse_order_selections` extracts only `choices`,
`service/bot/context/parsers.py`). A bot's commit can contradict its plan, its replies can
contradict both, and next phase it remembers none of it. The TDD promises "an internal diary
of plans and reasoning (updated across the phase loop)" in private context (**Context**,
`TECHNICAL_DESIGN.md`) and diary rows keyed to member and phase for inspection (**Telemetry
and observability**) ‚Äî none implemented.

Status: **described in the TDD but not implemented**; the single biggest behavioural delta
from AI_Diplomacy after tactical annotations.

## Why it matters

Memory is what makes a bot an opponent rather than a per-phase move generator: keeping a plan
across the phase, holding a grudge next phase, referring back to a deal in chat. It is also
the inspection surface the TDD wants for tuning ("read a bot's diary, plans‚Ä¶ as a phase
unfolds"). Gap 08 (relationships/goals) and Gap 09 (negotiation agenda) both persist their
state through the write path this doc creates.

Depends on Gap 06 (per-bot member identity), Gap 04 (the reasoning field that seeds entries),
Gap 05 (previous-phase order history as retrospective input). Feeds Gaps 08 and 09.

## Proposed approach

**Model.** In `service/bot/models.py`:

```python
class DiaryEntryQuerySet(models.QuerySet):
    def for_member(self, member):
        return self.filter(member=member)

    def recent_for_context(self, member, limit=12):
        return self.for_member(member).order_by("-created_at")[:limit]


class DiaryEntry(BaseModel):
    objects = DiaryEntryManager()
    member = models.ForeignKey("member.Member", on_delete=models.CASCADE, related_name="bot_diary_entries")
    phase = models.ForeignKey("phase.Phase", on_delete=models.CASCADE, related_name="bot_diary_entries")
    kind = models.CharField(max_length=20, choices=DiaryKind.KIND_CHOICES)
    body = models.TextField()
    llm_call = models.ForeignKey("bot.LLMCall", null=True, on_delete=models.SET_NULL, related_name="diary_entries")
```

`DiaryKind` in `service/bot/constants.py`: `PLAN`, `COMMIT`, `RETROSPECTIVE`. The `llm_call`
FK links each entry to the call that produced it, per the TDD's inspection design (Gap 02).
FKs point from `bot` into game-side models, same direction as `LLMCall`.

**Zero extra LLM calls ‚Äî entries ride existing ones.** This is the core cost decision, and
where we diverge from AI_Diplomacy's three dedicated diary calls per phase:

- The **plan** call's JSON (Gap 04) gains a `diary` field: two-to-four sentences of intent
  ("what I'm trying to do this phase and with whom"). Stored as `kind=PLAN`. The `reasoning`
  field remains free-form working-out and is *not* stored.
- The **plan** call also gains a `retrospective` field, produced because the prompt now
  contains the previous phase's resolved orders (Gap 05) and recent chat: "what happened last
  phase versus what you expected ‚Äî note kept and broken promises". Stored as
  `kind=RETROSPECTIVE` against the previous phase. This folds AI_Diplomacy's dedicated
  phase-result betrayal call into an existing call.
- The **commit** call's JSON gains a `diary` field noting final intent and any change from the
  plan. Stored as `kind=COMMIT`.
- Reply calls store nothing ‚Äî chat is already persisted as messages, and per-message diary
  writes would be noise.

Tasks in `service/bot/tasks.py` write entries right after parsing, inside the task
(background, not request cycle). Writes are best-effort: a missing `diary` key degrades to no
entry, never a task failure.

**Feed it back.** `ContextBuilder` gains `with_diary(entries)` rendering a private block:

```
Your diary (recent entries, oldest first):
  [Spring 1901, Movement ‚Äî plan] Open west; sound out France about Belgium.
  [Spring 1901, Movement ‚Äî commit] Went north instead; France evasive.
  [Spring 1901, Movement ‚Äî retrospective] France took Burgundy as feared‚Ä¶
```

`fetch_context` is unchanged (diary is bot-app data, read directly via the QuerySet in the
task ‚Äî the player-via-API boundary governs *game* data, and the TDD explicitly designs bot
reasoning as bot-app rows). Plan and commit prompts include the recent window (12 entries ‚âà
600‚Äì900 tokens); the reply prompt includes only the current phase's `PLAN` entry ‚Äî the agenda ‚Äî
mirroring AI_Diplomacy's latest-phase-only trick for order prompts
(`get_latest_phase_diary_entries`).

**Consolidation: sliding window now, summarization later.** `recent_for_context(limit=12)` is
a hard window (~4 phases √ó 3 entries); older entries persist in the table for inspection but
leave the prompt. AI_Diplomacy's yearly LLM consolidation is deferred: it costs one call per
bot per game-year and mainly matters for very long games. The window keeps context flat-cost
forever; if playtesting shows bots forgetting formative early-game events, add a consolidation
task then (design note kept in open questions).

## Cost impact

Zero additional LLM calls. Output grows by the `diary` + `retrospective` fields (~100‚Äì250
tokens on plan, ~50‚Äì100 on commit); input grows by the diary window (~600‚Äì900 tokens) on plan
and commit and ~80 tokens (agenda only) on replies. All diary content is private context ‚Äî
uncacheable by design ‚Äî so this is real marginal input on every call; the fixed window is
what keeps it constant per phase regardless of game length, honouring the fixed per-phase
budget. Rough net: +1.5‚Äì2k input tokens per bot-phase across the two critical calls.

## Scope boundaries

Out of scope here:

- Goals and relationships ‚Äî Gap 08 (separate rows, updated via the same plan call).
- The negotiation agenda's *use* (initiating messages) ‚Äî Gap 09; this doc only stores the
  plan entry Gap 09 reads.
- LLM consolidation of old entries ‚Äî deferred (open question).
- Any staff inspection UI ‚Äî the rows are Metabase/admin-readable; UI is future work.
- Exposing diary via the public API ‚Äî never; it is hidden information.

## Testing notes

In `service/bot/tests.py`:

- Plan task with a mocked `LLMClient` returning `{"reasoning": ..., "diary": "...",
  "retrospective": "...", "choices": [...]}`: assert one `PLAN` entry on the current phase,
  one `RETROSPECTIVE` on the previous phase (using a two-phase fixture via `phase_factory`),
  both FK-linked to the recorded `LLMCall`.
- Missing `diary` key ‚Üí orders still submitted, no entry, no error.
- Finalize task ‚Üí `COMMIT` entry created; running finalize twice (idempotence test mirrors
  `test_finalize_does_not_double_toggle_when_confirmed`) does not duplicate entries ‚Äî guard
  with `get_or_create` on `(member, phase, kind)` or an explicit existence check.
- `TestContextBuilder`: `with_diary` renders oldest-first with phase/kind tags;
  `recent_for_context` returns at most the limit, newest phases retained.
- Prompt-feedback test: second plan call in a two-phase game (pattern from
  `service/integration/test_bot.py::test_bot_can_play_the_next_phase`) receives the first
  phase's entries in its prompt (assert via captured prompt).

## Open questions

- Uniqueness: is one entry per `(member, phase, kind)` right, or should a re-planned phase
  append? Recommend unique-with-overwrite for PLAN/COMMIT (latest intent wins) ‚Äî needs
  confirmation once mid-phase replanning (deferred in TDD) is revisited.
- Window size 12: tune from transcripts; the trade is directly tokens-vs-memory-depth.
- When consolidation eventually lands, should it run on the bot's model or a cheaper one?
  AI_Diplomacy uses the agent's own client (`run_diary_consolidation` uses `agent.client`);
  on our budget a consolidation call would use the same single configured model ‚Äî flagged for
  whenever it is picked up.
